# 面试问题抽取提示词 v4

## 任务

从一篇牛客面经文章中抽取“可以直接拿来练习回答”的技术面试题。

目标问题风格参考题库 card 的 `question` 字段:

- `线上慢 SQL 怎么排查?`
- `Redis 热 key 怎么处理?`
- `JDK 1.8 ConcurrentHashMap 的底层实现是什么?`
- `Java 中 == 和 equals 有什么区别?`
- `TCP 三次握手的过程是什么,为什么不是两次?`

也就是说,`question` 必须是一个独立、清晰、可回答的单句问题。

## 输入

```text
方向关键词:{{query}}
文章标题:{{title}}
文章正文:
{{text}}
```

## 输出

只返回 JSON 对象,不要 markdown 代码块、注释或散文。

如果是面经并且能抽到问题:

```json
{
  "isInterview": true,
  "questions": [
    {
      "question": "Redis 分布式锁如何实现?",
      "category": "Redis",
      "difficulty": "medium",
      "evidence": "Redis分布式锁如何实现？",
      "confidence": 0.9
    }
  ]
}
```

如果不是面经,或没有可独立练习的技术问题,或(有方向时)全文没有与方向相关的技术题:

```json
{
  "isInterview": false,
  "questions": []
}
```

## 方向约束(优先于“尽量多抽”)

输入里的「方向关键词」决定本次练习范围:

- 当方向关键词为 `面经` 时: 不施加方向过滤,可抽取文中全部符合下方类别与质量规则的技术面试题。
- 当方向关键词为其他任何值(例如 `mysql`、`redis`、`kafka`、`java`) 时: **只输出与该方向直接相关的技术面试题**。这是硬性要求,不是建议。

「与方向直接相关」的判定:

- 题目考查的知识点、组件、框架、中间件、工具链,与方向关键词属于同一技术主题,或是其子主题/常见面试子考点。
- 示例: 方向 `mysql` — 索引、慢 SQL、事务隔离、锁、MVCC、主从复制; 方向 `redis` — 缓存、热 key、持久化、集群、分布式锁; 方向 `kafka` — 分区、副本、ISR、消费组、offset、幂等与顺序性; 方向 `java` — JVM、集合、并发、Spring 等 Java 生态考点。
- 同一篇面经里若还出现了与当前方向**无关**的其他技术题(例如方向是 `kafka` 却写了 MySQL 索引、Java 集合),**一律不要输出**,即使原文写得很清楚。
- 宁可少抽、甚至返回空列表,也不要为了凑数输出偏题问题。

若全文没有符合当前方向的技术题,返回 `isInterview: false` 且 `questions: []`(不要退而求其次输出无关技术题)。

## 抽题规则

- 只抽技术面试题、笔试题、手撕算法题、系统设计/项目追问。
- 不抽自我介绍、反问、薪资、流程、地点、公司评价、情绪表达、求助内容。
- 每个 `question` 必须是独立问题,不能依赖“上面这个”“这个项目”等上下文。
- 不要输出带现场口语上下文的问句:
  - 不要写 `结合你的经历...`
  - 不要写 `除了这个还有什么...`
  - 不要写 `这个怎么做`
  - 不要写 `你实操过...吗`
  - 要改写成可独立练习的标准问句
- 原文是关键词时,要改写成完整问句:
  - `线程池的参数` -> `线程池有哪些核心参数,分别有什么作用?`
  - `Docker拉取镜像与创建容器的命令` -> `Docker 如何拉取镜像并创建容器?`
  - `MySQL 回表` -> `什么是 MySQL 回表,如何避免?`
- 原文是追问式表达时,改写成知识点主问句:
  - `结合你的经历，说说MySQL数据库有哪些调优方式？` -> `MySQL 数据库有哪些常见调优方式?`
  - `除了缓存击穿，还有什么问题？` -> `Redis 缓存常见问题有哪些,分别如何处理?`
- 原文已经是清晰问句时,尽量保留原句。
- 同义或重复问题合并。
- 最多抽 12 个问题。

## category

category 必须严格从下面选择一个:

- `Redis`
- `Java`
- `MySQL`
- `计网`
- `计系统`
- `Agent`

无法归类到这 6 类的问题直接丢弃,不要硬填。

## difficulty

只能使用:

- `easy`
- `medium`
- `hard`

## confidence

0 到 1。原文明确出现的问题给高分; 需要改写但依据清楚的给中高分; 推测性问题不要输出。
