# Design Review

更新时间：2026-05-11

## 总体评分

当前项目实现评分：**7/10**

它已经是一个能用的本地训练台，不是 demo。主闭环完整，反馈区也有深度。但作为“面经训练产品”，现在最弱的是题池治理和信息架构：抓题能力强了以后，用户会被候选题淹没。

## 分项评分

| 维度 | 分数 | 判断 |
|---|---:|---|
| 红线闭环 | 8.5/10 | 导入、抓题、作答、评分、重答、保存卡片都通了 |
| 信息架构 | 6/10 | 现在仍是题目库、练习台、卡片库三大块，缺少“待确认 / 练习中 / 已沉淀 / 已隐藏”这种工作流分层 |
| 题池治理 | 4.5/10 | 只有单条忽略和 purge，缺少批量处理、重复识别、accepted 优先等关键治理能力 |
| 练习体验 | 8/10 | 作答、历史 attempt、best attempt、delta、反馈 tab 都不错 |
| 反馈呈现 | 7.5/10 | 信息丰富，但反馈内容很重，长反馈区块可能让用户不知道下一步先改什么 |
| 卡片沉淀 | 7.5/10 | 保存链路明确，重复卡片有覆盖确认，但缺少“已有相近卡片”提示 |
| 状态与空态 | 7/10 | 有空态和错误提示，但状态语义还没转成产品工作流 |
| 可扩展到 Phase B | 6.5/10 | attempts / scores 数据结构够用，但画像入口和只读聚合还没实现 |

## 主要缺陷

### P0：题池没有“整理模式”

现在题卡直接铺在 grid 里，用户可以搜索、分类、忽略，但没有一个明确动作叫“整理这批新题”。

结果是：抓取越强，题池越容易变成堆积场。

### P0：缺少 accepted 的产品地位

代码里有 `candidate / accepted / ignored / duplicate / mastered`，但 UI 主路径还没有把 `accepted` 当作真正的“可练题”。

现在 `candidate` 和 `accepted` 在体验上差异不够明显。

### P0：重复题目前只是 schema 状态，不是体验能力

有 `duplicate` 状态，但没有疑似重复提示，也没有处理动作。

对面经题库来说，这是很关键的，因为同一问题会以不同问法反复出现。

### P1：候选题浏览不应继续无限 grid

推荐方向：

```text
Tabs + 每页 N 题 + 当前筛选状态
```

第一版不建议做复杂多页面路由或虚拟滚动。先用同页 tabs + 分页验证题池治理工作流。

### P1：confidence 现在会误导用户

UI 里现在显示 `置信度 xx%`，练习页还用 confidence 判定“可练 / 需确认”。这会让用户以为它是科学质量分。

设计上应该把 confidence 降级成弱信号，甚至第一版可以隐藏小数，只展示“待确认 / 建议练习 / 用户已接受”。

### P1：反馈区信息太满，缺少“下一步”聚焦

反馈 tab 很完整，但用户每次评分后最该看到的是：

```text
这次最大问题是什么 -> 下一版怎么改 -> 是否保存 / 重答
```

现在信息丰富，但行动优先级还可以更强。

## 下一轮优先级

1. 题池 tabs：`待确认 / 练习中 / 已沉淀 / 已隐藏`
2. 批量选择 + 批量忽略
3. 疑似重复提示 + 标记 duplicate
4. accepted 优先：练习入口默认服务 accepted
5. confidence 降级：不再显示百分比，改成弱提示或隐藏
6. Phase B-lite 只读画像入口

## 设计结论

当前项目不是缺功能，而是缺“题池从原料变成练习计划”的中间层。

如果把 P0 题池治理做好，项目会从 **7/10 的功能型工具** 变成 **8.5/10 的个人训练工作台**。
